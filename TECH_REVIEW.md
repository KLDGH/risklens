# RiskLens — Architecture and Implementation Reference

**Status:** Reference document. Describes how the system is built, what choices were made, and where the architectural seams lie.
**Audience:** Engineers and quants who want to understand the implementation in detail rather than read the user-facing FAQ.
**Scope:** Data model, computation pipeline, frontend architecture, quantitative methods, operational profile, scalability characteristics, and known limitations.

---

## 1. What this system actually is

RiskLens is a static-site risk dashboard with a batch-computed data layer. The user-facing surface is a single-page React application; the data layer is a single JSON document regenerated by a daily Python batch job. There is no application server, no live database, no message queue, and no cache between the daily compute and the rendered UI. This is intentional and constrains every other architectural choice in the system.

The dashboard exposes per-asset and per-portfolio tail-risk metrics across a configurable universe of liquid ETFs and mutual funds. The computational core is a Python module (`backend/risk_engine.py`) that implements five distinct VaR models (Historical Simulation, EWMA, GARCH(1,1), GJR-tGARCH, and EVT via Generalized Pareto Distribution), Expected Shortfall under each model, the Hill estimator for tail-index alpha, an EWMA-covariance Component VaR decomposition, Kupiec unconditional coverage and Christoffersen independence backtesting, and several scenario aggregation functions (historical replay, hypothetical shock-vector application). A live signal layer pulls Treasury yield curve data and applies the Estrella-Trubin probit model to produce a current US recession probability; intraday SPY-TLT correlation is computed from 5-minute and 15-minute bars over a 60-day rolling window.

Three portfolio modes ship in the public version: a synthetic 60/30/8/2 multi-asset mix, the Vanguard Target Retirement 2055 fund's underlying ETF allocation, and the American Funds Target 2055 fund's underlying mutual-fund allocation. The same backend pipeline runs once per mode and emits a unified JSON keyed by mode identifier; the frontend toggle swaps which mode's data drives the UI without re-fetching.

The system has been built deliberately to make every quantitative output traceable to its inputs. The methodology is open in `backend/risk_engine.py` and documented in the FAQ; the JSON contract is the canonical interface between compute and render and is not hidden behind any API layer; every configuration value (window lengths, lambda decay, EVT threshold percentile, scenario shock vectors) is a Python constant in source rather than an opaque service parameter.

## 2. Stack and dependencies

The Python compute stack is `pandas`, `numpy`, `scipy`, `arch` (Kevin Sheppard's GARCH library — the only non-from-scratch piece of the methodology), and `yfinance` (the data adapter). Python 3.11+. There is no database, no ORM, no web framework on the backend; the entire compute is a sequence of pure functions over pandas DataFrames terminating in a `json.dump` call.

The frontend stack is React 18, Vite as the bundler, and Recharts for the charts. CSS is plain (no Tailwind, no styled-components). State management is component-local React state — there is no global state library, no context providers beyond what trivial prop-drilling avoids. The bundle is approximately 590 KB compressed.

Hosting is GitHub Pages. The data refresh is a scheduled GitHub Actions workflow with a daily cron trigger.

The dependency footprint is intentionally small. Each library was added because it solves a specific problem the from-scratch alternative would solve worse: `arch` because reimplementing GARCH MLE with proper convergence handling is a multi-week project that produces inferior code to Sheppard's; `yfinance` because building a Yahoo Finance scraper from scratch is brittle and pointless; Recharts because it produces good-looking SVG charts with minimal code. Everything else (HS, EWMA, EVT, Hill, Kupiec, Christoffersen, Component VaR, scenario aggregation, intraday correlation) is implemented from first principles in `risk_engine.py`.

## 3. Architecture and data flow

The daily flow is: scheduled GitHub Actions runner spins up an Ubuntu container, installs Python 3.11 and Node.js 20, runs `pip install -r backend/requirements.lock.txt`, executes `python backend/run.py` (which fetches market data, computes all metrics, writes the JSON), runs `npm install && npm run build` in `frontend/`, and deploys the resulting static bundle to GitHub Pages. End-to-end wall-clock time is approximately 4 minutes; the Python compute itself is 60-90 seconds depending on whether the GARCH backtest cache is being regenerated.

The frontend is a single-page React application. On load it issues one fetch to `data/risk_output.json` and renders the entire dashboard from the resulting object. There are no subsequent network calls during normal operation. The component graph is shallow — the `App` component holds the data and the active portfolio mode in local state, and every section component (`RiskTable`, `PortfolioRiskChart`, `BacktestPanel`, `ScenarioPanel`, `HistoricalChart`, `CorrelationChart`, `MultiWindowCorrelationChart`, `IntradayCorrelationChart`) receives the relevant slice of the JSON as props. Cross-component state is minimized: each section manages its own UI state (chart toggles, expanded/collapsed panels, hovered cells) entirely locally.

The single critical path through the system is the daily Python batch. It runs sequentially through five logical phases. (1) Data acquisition — fetch end-of-day prices, intraday bars, VIX, S&P 500 history, Treasury yields. (2) Per-asset risk computation — for each ticker in the active universe, compute all five VaR estimates, all five ES estimates, the Hill tail index, and the EWMA-VaR percentile rank ("Risk gauge"). (3) Per-portfolio aggregation — for each portfolio mode, build the weighted return series and run the same computation on it. (4) Scenarios and backtests — historical scenarios replay actual price data over defined date ranges; hypothetical scenarios apply analyst-estimated shock vectors; backtests run rolling out-of-sample VaR forecasts and compute Kupiec/Christoffersen statistics. (5) Multi-window correlation, intraday correlation, recession probability — each independent and quick.

Phases (1) through (4) are pipelined sequentially because their compute is dominated by tractable single-machine math; phase (5) could in principle run in parallel with the others but the savings would be marginal (~5 seconds out of ~60). The pipeline is intentionally simple: one process, one machine, no orchestration framework. If it fails, the previous day's JSON remains served and the failure is visible in GitHub Actions logs.

## 4. Data model — the JSON contract

The system's most important interface is the schema of the JSON document the daily batch writes. Both the data producer and the UI consumer treat this document as the source of truth, and feature additions are expressed as schema extensions rather than sidecar API endpoints.

Top level:

```
{
  "generated_at":          ISO8601 UTC timestamp,
  "data_as_of":            "YYYY-MM-DD" — last US market close represented,
  "default_mode":          mode_id selected on initial render,
  "portfolios":            { mode_id: per-mode bundle },
  "sp500_history":         per-year aggregate stats (1990 onward),
  "correlation_history":   weekly samples of average pairwise correlation across a fixed basket,
  "multi_window_corr":     SPY × bond rolling correlations at 20d/60d/252d windows,
  "intraday_corr_history": 5m/15m intraday SPY-TLT correlation, last 60 trading days
}
```

Per-mode bundle:

```
{
  "label":        display name,
  "description":  paragraph string,
  "weights":      { ticker: weight (sum to 1.0) },
  "assets":       [ ...per-asset rows..., portfolio summary row at end ],
  "scenarios":    [ ...scenario records, type = historical | hypothetical ],
  "risk_history": [ { date, var } ],   // weekly samples
  "backtests":    [ ...one record per VaR model... ]
}
```

Per-asset row:

```
{
  "ticker":            string,
  "name":              string,
  "last_price":        number,
  "last_return_pct":   number,
  "var_hs":            number, "es_hs": number,
  "var_ewma":          number, "es_ewma": number,
  "var_garch":         number, "es_garch": number,
  "var_tgarch":        number, "es_tgarch": number,
  "var_evt":           number, "es_evt": number,
  "tail_index":        number,
  "mean_var":          number,    // simple average across the 5 VaR models
  "risk_level":        number,    // percentile rank in [0, 1] vs trailing 504-day history
  "var_trend":         "up" | "down" | "flat",
  "exception_count":   int,       // EWMA-VaR breaches over trailing 504 days
  "exception_rate":    number,    // count / 504, in percent
  "component_var":     number     // contribution to portfolio EWMA VaR (per-asset; absent on portfolio row)
}
```

Scenario record:

```
{
  "id":             string identifier,
  "name":           display name,
  "desc":           one-sentence description,
  "type":           "historical" | "hypothetical",
  "start":          "YYYY-MM-DD" (historical only),
  "end":            "YYYY-MM-DD" (historical only),
  "portfolio_pnl":  total scenario P&L on $100 (percent),
  "coverage_pct":   fraction of weight actually covered (handles tickers that didn't exist in the date range),
  "asset_returns":  { ticker: percent_return },
  "contributions":  { ticker: percent_contribution_to_portfolio_pnl },

  // Hypothetical-only:
  "probability_live":    { name, value, value_label, context, url } | absent,
  "probability_sources": [ { name, url, note } ] | absent
}
```

Backtest record:

```
{
  "model":            "HS" | "EWMA" | "EVT" | "GARCH" | "tGARCH",
  "exceptions":       int (count over 504-day eval window),
  "expected":         number (504 × p),
  "rate_pct":         number (observed rate, percent),
  "expected_pct":     number (expected rate, percent — usually 1.0),
  "kupiec_p":         number,         // Kupiec unconditional coverage p-value
  "christoffersen_p": number,         // Christoffersen independence p-value
  "verdict":          "CALIBRATED" | "UNDER-EST" | "OVER-CONSERV" | "CLUSTERED",
  "eval_dates":       [ "YYYY-MM-DD", ...504 ],
  "violation_dates":  [ subset of eval_dates where the model's VaR was breached ]
}
```

The contract has held up across several rounds of feature additions. Adding a new analytical computation cleanly maps to either (a) a new field on an existing record, (b) a new top-level subtree, or (c) a new sub-record list under an existing record. The frontend treats unknown fields as no-ops and missing optional fields as feature-not-available, so producer and consumer can evolve at slightly different rates without a coordinated deploy.

## 5. Methodology

The methodology section is the longest and the most consequential — every other architectural choice in the system serves the goal of making these computations transparent and auditable.

**Historical Simulation VaR** is the empirical p-th quantile of the trailing 1000 daily returns. The choice of 1000 days as the lookback window is intentional: at p = 0.01, the window contains exactly 10 observations in the loss tail, which is the smallest count at which the empirical quantile is reasonably stable. Smaller windows (252 days, 504 days) yield 2-5 tail observations and produce a noisy estimate; larger windows (2000+ days) start to smear across volatility regimes. ES is the average of all observations at or beyond the VaR threshold. No distribution assumption.

**EWMA VaR** uses Riskmetrics-style exponentially weighted volatility with λ = 0.94, the original 1996 J.P. Morgan default and still the industry-standard choice. The recursion `var_t = (1-λ) × r_t² + λ × var_{t-1}` is run forward through the return series; the most recent value is the conditional variance forecast. VaR and ES are computed under a normal-distribution assumption around mean zero. The advantage of EWMA over a rolling-window historical variance is that recent observations are weighted more heavily, allowing the forecast to react to volatility regime changes within days rather than weeks. The disadvantage is the normal-distribution tails — at the 1% level, EWMA systematically under-estimates fat-tail events.

**GARCH(1,1)** uses Bollerslev (1986) conditional variance dynamics: `var_t = ω + α × r_{t-1}² + β × var_{t-1}`. Parameters are estimated by maximum likelihood via the `arch` library. The one-step-ahead conditional variance forecast is fed into a normal-distribution VaR formula. The MLE has reliability issues for return series with extreme outliers; the implementation falls back to EWMA on convergence failure.

**GJR-tGARCH** extends GARCH with an asymmetric term that gives extra weight to negative shocks: `var_t = ω + α × r_{t-1}² + γ × r_{t-1}² × I(r_{t-1} < 0) + β × var_{t-1}`. Empirically this captures the "leverage effect" — volatility rises more after losses than after equivalent gains. Same MLE setup, same fallback to EWMA on convergence failure.

**EVT** uses the peaks-over-threshold approach. Losses exceeding the 90th-percentile threshold are fit to a Generalized Pareto Distribution; the VaR at confidence level p is computed analytically from the GPD parameters. This is the only model in the suite that does not assume normal tails. In fat-tail regimes EVT captures losses the four parametric models miss; the trade-off is that EVT is sensitive to threshold choice and tends toward over-conservatism on cleaner data.

The **Hill estimator** for tail-index alpha is computed independently from EVT. Losses are sorted descending; alpha equals `k / Σ ln(L_i / L_k)` where the sum runs from i=1 to k-1 and k is roughly √n (with a minimum of 10). Lower alpha indicates fatter tails. Broad equity indices typically sit around 3-4; individual stocks 2-4; high-volatility assets like crypto and gold often below 3.

**Component VaR** decomposes the portfolio's parametric VaR into per-holding contributions using the EWMA covariance matrix. The mathematical identity is `Component_i = w_i × (Σw)_i / σ_p × z × portfolio_value`, where Σ is the EWMA covariance estimate, w is the weight vector, σ_p is portfolio volatility, and z is the standard normal quantile. The sum of components exactly recovers the portfolio's parametric VaR. Negative components indicate hedges — assets whose covariance with the rest of the portfolio reduces total risk.

**Backtesting** uses Kupiec (1995) unconditional coverage and Christoffersen (1998) independence tests, both applied to the model's daily violation indicator over a 504-day evaluation window. For each day in the eval window the VaR forecast is computed using only the prior 1000 days of returns (strict out-of-sample). The Kupiec test compares the observed exception rate to the expected p; the Christoffersen test asks whether exceptions are statistically independent across consecutive days. A directional verdict (`CALIBRATED`, `UNDER-EST`, `OVER-CONSERV`, `CLUSTERED`) is computed from the two p-values; this surfaces *how* a model is mis-calibrated rather than collapsing failure modes into a binary pass/fail.

GARCH and GJR-tGARCH backtests are computationally heavier than the others because the MLE refit is required at each rolling step. The implementation uses warm-started MLE — each refit's starting parameter values are seeded from the previous fit's converged values, which typically reduces convergence iterations from 20-50 down to 3-10. The resulting backtest takes about 20-30 seconds per portfolio per model rather than several minutes. Results are cached separately at `backend/cache/garch_backtests.json` and refreshed only when explicitly requested via `RISKLENS_FULL_BACKTEST=1`, keeping the daily refresh fast.

**Historical scenarios** replay actual prices over a defined date range. The portfolio's P&L is computed by re-weighting available tickers (re-normalizing if some tickers didn't exist in the date range) and aggregating asset-level returns. The result is fully data-driven; no shock vector or model assumption is involved.

**Hypothetical scenarios** apply analyst-estimated per-asset shock vectors. Each scenario has a `shocks` dict mapping ticker to expected percentage move conditional on the scenario occurring. Aggregation is a weighted sum: `portfolio_pnl = Σ shocks_i × normalized_weights_i`. This is a per-asset specification (not factor-driven) and the linearity is intentional — it makes the contribution attribution exact and visible. The trade-off is that the shock vectors encode an implicit correlation assumption rather than computing it; for a more rigorous treatment see §8 (factor decomposition is listed as a known gap).

**Live recession probability** uses the Estrella-Trubin (2006) NY Fed model: `P(recession in 12mo) = Φ(-0.5333 - 0.6330 × spread)`, where spread is the 10Y-3M Treasury yield difference in percentage points. The two yield series are pulled from Yahoo Finance (`^TNX` and `^IRX`); the most recent values are plugged into the formula. The output is exposed on the US Recession hypothetical scenario card.

**Intraday correlation** uses 5-minute and 15-minute bars from yfinance (the only intraday data available without a paid feed; constrained to the last 60 days). For each trading day, log returns of SPY and TLT are computed at the chosen frequency and the within-day Pearson correlation is calculated. Each daily value is its own statistically meaningful estimate (n ≈ 78 at 5-minute, n ≈ 26 at 15-minute) rather than a single point in a rolling window. The signal is the consecutive same-sign streak, which under reasonable null hypotheses is a sharp regime-shift indicator.

## 6. Frontend rendering

The frontend is structured around analytical sections, each implemented as a self-contained component receiving a slice of the JSON. Sections are: `RiskTable` (the per-asset risk grid with portfolio summary row), `PortfolioRiskChart` (daily EWMA VaR area chart with crisis annotations), `BacktestPanel` (Kupiec/Christoffersen verdict table with VaR exception calendar), `ScenarioPanel` (historical and hypothetical stress test cards), `HistoricalChart` (per-year S&P 500 risk bars with VIX overlay), `CorrelationChart` (60-day rolling cross-asset correlation), `MultiWindowCorrelationChart` (multi-window stock-bond correlation with bond-proxy toggle), and `IntradayCorrelationChart` (calendar/bar view of intraday SPY-TLT correlation with sampling-interval toggle).

A reusable `Section` wrapper component renders the section title and a collapsible "About this section" toggle that reveals the section description and curated literature references. This keeps the page compact by default while preserving methodological context one click away.

The `CalendarHeatmap` component is a reusable GitHub-contribution-style calendar grid used for both the intraday correlation view and the VaR exception calendar in the backtest panel. The component takes a flat array of `{date, value, ...}` records and a color function, computes a Mon-Fri × N-week grid, and renders cells with shared hover state. Reuse across two distinct sections kept the implementation simple.

Cross-component state is intentionally minimized. The active portfolio mode lives at the `App` level; everything else is local to its section. This avoids the entire class of state-synchronization bugs that would emerge from a shared store, at the cost of some prop-drilling that has not yet become painful.

## 7. Operational profile

The daily refresh runs unattended via the GitHub Actions cron trigger at 06:30 UTC weekdays. End-to-end wall-clock time is approximately 4 minutes (including Python install, Node install, JSON regeneration, frontend build, and GitHub Pages deploy). The Python compute itself is 60-90 seconds.

Failure modes observed in production: yfinance occasionally returns malformed responses for individual tickers (handled with per-ticker try/except in `fetch_data.py`); Yahoo Finance occasionally throttles or temporarily blocks requests (no observed impact at our request volume of ~60 daily requests); GARCH MLE occasionally fails to converge for high-volatility assets (handled with EWMA fallback). None of these have caused a complete daily-refresh failure in the observed period.

Storage: the JSON output is approximately 800 KB uncompressed and gzips to ~80 KB on the wire. The cached GARCH backtest file (`backend/cache/garch_backtests.json`) is approximately 15 KB. The entire repository including code, JSON, and cache is well under 5 MB.

Compute profile: the Python pipeline runs on a single machine, single process. Memory usage peaks at approximately 200 MB during the GARCH backtest (when fitting on 1000-day windows for 504 evaluation days). CPU usage is single-threaded; embarrassingly parallel components (per-asset risk, per-portfolio aggregation) are not currently parallelized because the serial time is acceptable.

Monitoring: GitHub Actions surfaces job success/failure and execution time as part of its standard UI. The frontend reads a `data_as_of` field from the JSON and displays it in the top right of the page; if the daily refresh has been failing for multiple days the displayed date will be stale, providing a passive freshness indicator visible to users.

There is no application server, no live database, no cache layer, no CDN beyond GitHub Pages's defaults. The system has no operational state outside the JSON file and the GARCH backtest cache.

## 8. Scalability characteristics

The compute scaling is well-defined and presents no surprises through the regime where it might be tested.

**Per-asset risk computation** is O(N) in the number of unique tickers. The dominant per-asset cost is GARCH MLE fitting, approximately 0.5-1 second per asset on commodity hardware. At N = 100 tickers the per-asset pipeline takes about 1 minute; at N = 1000 tickers, about 15-30 minutes serial. The pipeline is embarrassingly parallel — `multiprocessing.Pool` or `joblib.Parallel` would scale this near-linearly with worker count.

**Per-portfolio computation** is O(P) in the number of distinct portfolios. The dominant per-portfolio cost is the rolling backtest at approximately 1-2 minutes per portfolio (with GARCH cached) or 5-8 minutes (with full GARCH refit). At P = 10 portfolios this fits comfortably in a daily window; at P = 100 it becomes the dominant cost and benefits from parallelization.

**Cross-asset correlation matrix** is O(N²) in tickers but trivially fast at any reasonable N. A 1000×1000 EWMA covariance is on the order of milliseconds in numpy. Memory is also O(N²) — 8 MB at N=1000 — which is negligible. Visual rendering of cross-asset correlation matrices loses interpretability past N ≈ 30, but the underlying compute scales fine.

**JSON document size** is O(N × P) for per-asset records times portfolios. The current public version (28 unique tickers × 3 portfolios) produces an 800 KB document. At 100 tickers × 10 portfolios the document would be approximately 10 MB uncompressed (~1 MB gzipped on the wire). At larger sizes the architecture would benefit from splitting the unified document into one-per-portfolio JSONs plus a small index document, with the frontend lazy-loading on portfolio selection. The frontend's mode-toggle pattern already maps cleanly to this — a mode switch already triggers a full re-render from a different data slice; replacing "different in-memory slice" with "different fetched JSON" is a small refactor.

**Frontend memory** scales with the size of the loaded JSON. At the current 800 KB, runtime memory after parsing is roughly 5 MB of JS heap. Modern browsers handle this with no issue.

**Network**: a single ~80 KB JSON fetch on page load, then no further network activity during normal operation. CDN caching is appropriate; cache-busting on the JSON URL via a content hash or query parameter handles the daily refresh.

## 9. Risks and limitations

**Methodology limitations**

The five-VaR-model approach is unusual; most risk dashboards pick one. The model-disagreement-as-signal philosophy is intentional but produces a denser visual surface than single-model dashboards.

The hypothetical scenario shocks are per-asset rather than per-factor. A factor-shock-propagation approach (Barra/MSCI style) would be more methodologically rigorous and more reusable across portfolios — the same "Taiwan invasion" factor shock vector would apply identically to any portfolio with the right factor loadings. This is the largest known methodological gap and is documented in the FAQ. Implementing it requires building a factor model layer with estimated factor returns and per-asset loadings, which is a multi-week project.

Backtesting covers HS, EWMA, EVT, GARCH, and GJR-tGARCH, all at 1-day VaR. Multi-period VaR (10-day, 1-month) is not implemented. Time-scaling under √N assumes normal returns and breaks for fat tails; a rigorous multi-period implementation would either fit models to multi-day overlapping returns directly or use a Monte Carlo path approach. Either is a meaningful add.

The intraday correlation signal is constrained to the last 60 days by yfinance's free-tier intraday window. There is no architectural barrier to extending it; the constraint is purely the data feed.

**Data limitations**

yfinance occasionally returns inconsistent data. The implementation uses forward-filling to handle weekends and missing days, which is correct for end-of-day analysis but would be wrong for any sub-daily use case.

The Treasury yield series (`^TNX`, `^IRX`) used for the recession probability has occasional inconsistent units; the implementation includes a heuristic to detect and correct this (`_last("^TNX")` divides by 10 if the raw value exceeds 30, accommodating Yahoo's intermittent reporting in tenths-of-a-percent vs percent).

Mutual fund holdings (for the Capital Group target-date mode) are static — the displayed weights are anchored to a published fact sheet rather than fetched live. A more rigorous implementation would fetch holdings from the fund company's data feeds; this hasn't been built because no free API exists.

**Architectural limitations**

The single-document JSON architecture has a soft ceiling around 5 MB on the wire; past that, loading time becomes user-visible and the frontend's parse-everything-up-front approach starts to feel slow. The mitigation (per-portfolio JSON split with lazy loading) is well-defined and easy to implement when needed.

The Python pipeline is serial. Parallelization is straightforward but adds operational complexity (more failure modes, harder log inspection); current scale doesn't justify it.

The frontend has no server-side rendering. Initial load shows a brief loading state before the JSON arrives; for slow connections this is visible. SSR is not architecturally hard but adds significant infrastructure (Node runtime, build pipeline complexity) for marginal benefit at current scale.

**Operational limitations**

No alerting beyond GitHub Actions' built-in failure notifications. If yfinance returns garbage data for a week, the dashboard would silently display garbage values — only a user looking at the displayed `data_as_of` and noticing it's stale would catch this. A simple "last successful refresh" health endpoint and a corresponding monitor would close this gap.

No audit logging. Anyone with the URL can view the dashboard; there is no record of who viewed what when. This is appropriate for a public OSS project and would not be appropriate for any deployment with confidential data.

## 10. Alternatives considered

**Vendor risk systems** (Bloomberg PORT, MSCI BarraOne, Aladdin, FactSet) — these are mature, factor-aware, regulator-trusted, and far more comprehensive than RiskLens. They are also expensive, designed for risk specialists, and proprietary in their methodology. RiskLens is complementary, not a replacement: a methodology-transparent, accessible-to-non-quants visualization layer that can sit alongside vendor systems.

**Open-source quant frameworks** (QuantLib, Pyfolio, vectorbt) — these provide more sophisticated risk math than RiskLens but no UI layer and a different scope. QuantLib is fundamentally a derivatives-pricing library where risk is a small subset; Pyfolio is backtest-focused; vectorbt is performance-tuned for systematic strategy backtesting. The bespoke implementation in `risk_engine.py` is appropriate for the scope.

**No-code BI tools** (Tableau, PowerBI, Sigma, Looker) — these are visualization layers, not computation layers. They can render numbers that came from somewhere else; they cannot recompute those numbers from first principles, surface model disagreement as a signal, or run their own Kupiec/Christoffersen tests. For methodology-explicit use cases they are the wrong tool.

**Server-side rendering and live data** — would require an application server, a database, a cache, a job queue, and corresponding operational infrastructure. The current architecture's choice to be batch-compute → static-artifact → static-render is a deliberate simplification that the use case justifies. Going live would be appropriate if intraday or sub-daily latency were a real requirement; it is not.

**Database-backed data layer** (Postgres, DuckDB, Parquet on S3) — would replace the single-JSON file with structured storage and per-record queries. At current scale this adds complexity without value; at 10x current scale it would become natural. The data model (§4) is already structured cleanly enough that a database migration would be largely mechanical.

**Different chart library** (D3, Plotly, Highcharts) — Recharts was chosen for its React-native API and acceptable default styling. D3 would offer more flexibility at significantly higher implementation cost; Plotly produces fine-looking charts but adds a heavy bundle; Highcharts has a commercial license. Recharts has been adequate; rewriting would not produce visible improvement.

## 11. Open technical questions

**Factor decomposition.** The largest known methodological gap. A rigorous implementation requires defining a factor universe (style, sector, country, macro), estimating factor returns over time, regressing per-asset returns on factor returns to obtain loadings, and rebuilding the scenario and risk attribution layers around factor space rather than asset space. Multi-week project.

**Multi-period VaR.** 1-day VaR is implemented; 10-day and 1-month are not. Time-scaling by √N is the simple but fat-tail-broken approach; rolling-overlap fitting is more rigorous. A choice point.

**Real-time intraday refresh.** Currently a batch artifact. Going intraday means rebuilding most of the engine and adding live data subscription infrastructure. Not architecturally hard but a significant scope expansion.

**Holdings ingestion for actively-managed funds.** The Capital Group mode uses static fact-sheet weights. A live ingestion path would need a holdings data source (none free), a parser, and a refresh cadence aligned with the fund company's disclosure cycle.

**Custom portfolio definitions.** Currently three modes are hardcoded. A "upload your CSV holdings" or "compose a portfolio interactively" feature would require persistent storage of user state and corresponding access control — both of which would push the architecture out of static-site territory.

**Confidence interval bands on charts.** The risk trajectory chart shows a single point estimate per day. A 95% CI band would be more honest about estimation uncertainty but requires either bootstrap resampling (computationally heavy) or a closed-form variance estimate (model-specific).

**Better alerting.** The current "monitor displayed data_as_of and notice if stale" approach is passive and depends on a human looking at the page. An active monitor that pings the JSON URL daily and alerts on staleness would be a small addition.

## 13. Reproducing a daily run

```bash
git clone <repo>
cd risklens

# Backend
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.lock.txt
python run.py
# Writes: ../frontend/public/data/risk_output.json

# Optional: regenerate the GARCH backtest cache (slow, ~5 minutes)
RISKLENS_FULL_BACKTEST=1 python run.py

# Frontend
cd ../frontend
npm install
npm run dev
# Open http://localhost:5173 (or whatever port Vite picks)
```

Production deploy is fully automated via the GitHub Actions workflow; no manual step is required after pushing to `main`.
