# RiskLens — Architecture and Implementation Reference

**Status:** Reference document. Describes how the system is built, what choices were made, and where the architectural seams lie.
**Audience:** Engineers and quants who want to understand the implementation in detail rather than read the user-facing FAQ.
**Scope:** Data model, computation pipeline, frontend architecture, quantitative methods, operational profile, scalability characteristics, and known limitations.

---

## 1. What this system actually is

RiskLens is a static-site risk dashboard with a batch-computed data layer. The user-facing surface is a single-page React application; the data layer is a single JSON document regenerated by a daily Python batch job. There is no application server, no live database, no message queue, and no cache between the daily compute and the rendered UI. This is intentional and constrains every other architectural choice.

The user-facing surface is organized into **three top-level tabs**:

1. **Portfolio Risk** — per-asset and per-portfolio risk metrics, model validation, stress tests. Responds to the portfolio-mode toggle.
2. **Market Context** — portfolio-independent reference charts (S&P 500 history, cross-asset and stock-bond correlation, intraday SPY-TLT correlation).
3. **Sector Spotlight** — single-asset deep-dive for sector ETFs (risk profile, Fama-French factor regression, three stacked anomaly detectors).

The computational core lives in `backend/risk_engine.py` (per-asset risk, portfolio aggregation, scenarios, backtesting, intraday correlation, QMLE polarization, anomaly detectors) and `backend/factor_models.py` (Fama-French + Momentum regression, OLS beta computation, Ken French data fetcher). A separate `backend/preprocess_holdings.py` module reads sponsor-disclosed xlsx/csv holdings files and emits a clean JSON for active-fund look-through.

**Five portfolio modes** ship in the public version:

| Mode | Construction | Tickers |
|---|---|---|
| `hypothetical` | Synthetic 60/30/8/2 mix from sector/asset-class ETFs | 14 |
| `tdf_2055` | Vanguard Target 2055 (VFFVX) underlying ETF allocation | 4 (VTI, VXUS, BND, BNDX) |
| `cg_2055` | American Funds Target 2055 (AAFTX) underlying mutual funds | 12 actively-managed AF funds |
| `cggo_active` | CGGO Look-Through basket — top 25 disclosed holdings, weights re-normalized | 25 underlyings + CGGO ref row |
| `dwld_active` | DWLD Look-Through basket — same architecture | 24 underlyings + DWLD ref row |

The same pipeline runs once per mode and emits a unified JSON keyed by mode identifier; the frontend toggle swaps which mode's data drives Tab 1 without re-fetching.

## 2. Stack and dependencies

**Backend (Python 3.10+):** `pandas`, `numpy`, `scipy`, `arch` (Sheppard's GARCH MLE library — the only non-from-scratch piece of the methodology), `yfinance` (Yahoo Finance adapter), `openpyxl` (xlsx parsing for sponsor holdings files, dev-only — CI reads the preprocessed JSON, never the xlsx).

**Frontend:** React 18 + Vite + Recharts. Plain CSS, no UI framework. Component-local state, no global store. ~640 KB bundle, ~190 KB gzipped.

**Hosting:** GitHub Pages via GitHub Actions. Daily cron at 06:30 UTC on weekdays.

**External free data sources:**
- Yahoo Finance via yfinance — daily EOD adjusted prices + 5m/15m intraday for the last 60 days.
- Ken French Data Library — daily Fama-French 5 factors + Momentum, public.

The dependency footprint is intentionally small. Each library was added because it solves a specific problem the from-scratch alternative would solve worse: `arch` because reimplementing GARCH MLE with convergence handling is a multi-week project that produces inferior code; `yfinance` because building a Yahoo scraper is brittle; Recharts because it ships SVG charts with minimal code; `openpyxl` because writing an xlsx parser is silly when there's one trustworthy library. Everything else (HS, EWMA, EVT, Hill, Kupiec, Christoffersen, Component VaR, scenario aggregation, intraday correlation, QMLE polarization, Page CUSUM, FF regression, OLS solver via numpy.linalg.lstsq) is implemented from first principles.

## 3. Architecture and data flow

Daily flow: GitHub Actions runner spins up Ubuntu, installs Python 3.11 + Node 20, runs `pip install -r backend/requirements.lock.txt`, executes `python backend/run.py` (fetches market data, computes all metrics, writes the JSON), runs `npm install && npm run build` in `frontend/`, deploys the static bundle to GitHub Pages. End-to-end wall-clock is roughly 4 minutes; the Python compute itself is 90-120 seconds (depending on whether the GARCH backtest cache is being regenerated and whether the FF factor fetch hits live or cache).

The frontend issues one fetch to `data/risk_output.json` on load and renders the entire dashboard from the resulting object. No subsequent network calls during normal operation. Component graph is shallow — `App` holds the active tab and the portfolio mode in local state; each section component receives the relevant JSON slice as props.

The Python batch is a single sequential pipeline that runs through these logical phases:

1. **Data acquisition** — fetch EOD prices for the master ticker union (all portfolio modes + bond proxies + anomaly-detector sector ETFs); fetch S&P 500 history, VIX, Treasury yields, intraday bars.
2. **Per-mode active-fund expansion** — read `backend/data/active_fund_holdings.json`, replace each active-fund mode's tickers + weights with its top-25 disclosed holdings (re-normalized over the modeled subset).
3. **Per-asset risk computation** — for each ticker in the active universe, compute all five VaR estimates, all five ES estimates, the Hill tail index, the EWMA-VaR percentile rank ("Risk gauge"), and the trailing-504-day exception count.
4. **Per-portfolio aggregation** — for each portfolio mode, build the weighted return series, run all five risk models on it, compute component VaR per holding via EWMA covariance, and compute the risk trajectory (daily EWMA VaR over the portfolio's available history).
5. **Scenarios** — historical replay over fixed date ranges with re-normalization for tickers that didn't exist; hypothetical shock-vector application.
6. **Backtesting** — Kupiec UC + Christoffersen IC on 504-day eval windows × 5 models × 5 portfolios. GARCH/tGARCH results read from cache by default; regenerated via `RISKLENS_FULL_BACKTEST=1`.
7. **Market context** — S&P 500 per-year aggregate stats, cross-asset rolling correlation (10 ETFs since 2007), multi-window stock-bond correlation across 4 bond proxies, intraday SPY-TLT correlation at 5m and 15m (last 60 days, with QMLE polarization estimator alongside naive realized correlation).
8. **Live signals** — Estrella-Trubin recession probability from the current 10Y-3M Treasury spread.
9. **Sector Spotlight views** — for each of 14 sector ETFs: full risk profile (same VaR/ES models from phase 3), beta vs SPY, Fama-French + Momentum regression, three detector time series (z-score, Page CUSUM, GARCH-residual) with flagged-date index.

Wall-clock breakdown: phases 1–5 ~30s; phase 6 ~5s (cached) or ~3 minutes (full GARCH regen); phase 7 ~10s; phase 8 ~5s; phase 9 ~30-45s.

## 4. Data model — the JSON contract

The system's most important interface. Both the producer and the UI consumer treat this document as the source of truth. Feature additions are schema extensions, not sidecar API endpoints.

**Top level:**

```
{
  "generated_at":          ISO8601 UTC timestamp,
  "data_as_of":            "YYYY-MM-DD" — last US market close represented,
  "default_mode":          mode_id selected on initial render,
  "portfolios":            { mode_id: per-mode bundle },
  "sp500_history":         per-year aggregate stats (1928 onward),
  "correlation_history":   weekly samples of average pairwise correlation across a fixed basket,
  "multi_window_corr":     SPY × bond rolling correlations at 20d/60d/252d windows,
  "intraday_corr_history": 5m/15m intraday SPY-TLT correlation, last 60 trading days, with both naive + QMLE estimators,
  "anomaly_views":         { tickers, names, data: { ticker: per-ticker anomaly bundle } }
}
```

**Per-mode bundle:**

```
{
  "label":        display name,
  "description":  paragraph string,
  "weights":      { ticker: weight (sum to 1.0) },
  "assets":       [ ...per-asset rows..., portfolio summary row at end ],
  "scenarios":    [ ...scenario records, type = historical | hypothetical ],
  "risk_history": [ { date, var } ],
  "backtests":    [ ...one record per VaR model... ],

  // Active-fund modes only:
  "is_active_fund_spotlight": true,
  "fund_ticker":              "CGGO" | "DWLD",
  "disclosed_weights":        { ticker: pct_of_fund },   // pre-normalization disclosed weight
  "coverage_meta":            { modeled_n, modeled_weight_pct, total_holdings, as_of },
  "fund_disclosure":          { ticker, fund_name, sponsor, mandate, as_of,
                                n_holdings, top10_concentration_pct,
                                top25_concentration_pct, holdings: [...] }
}
```

**Per-asset row:** unchanged — ticker, name, last_price, last_return_pct, var_hs/ewma/garch/tgarch/evt + es_*, tail_index, mean_var, risk_level, var_trend, exception_count, exception_rate, component_var.

**Anomaly view (per ticker on the Sector Spotlight tab):**

```
{
  "ticker": "XLF",
  "series": [
    { date, price, ret_pct, zscore, cusum_pos, cusum_neg, garch_resid }
  ],
  "anomalies": [
    { date, detectors: [...], ret_pct, zscore, cusum_pos, cusum_neg, garch_resid }
  ],
  "thresholds": { zscore, cusum, garch_resid },
  "params": { zscore_window, cusum_k, lookback_days },
  "risk_profile": { vol_60d_annualized_pct, var_hs/ewma/garch/tgarch/evt,
                    es_*, tail_index, risk_level, beta_spy_252d,
                    last_price, last_return_pct },
  "factor_model": {
    "model": "Fama-French 5 + Momentum (Carhart-extended)",
    "lookback_days": 252,
    "first_date", "last_date",
    "r_squared", "alpha_daily_pct", "alpha_annualized_pct",
    "alpha_tstat", "alpha_pvalue",
    "total_vol_annualized_pct",
    "factor_vol_annualized_pct",
    "idio_vol_annualized_pct",
    "factor_variance_share_pct",
    "loadings": [ { factor, label, beta, tstat, p_value, significant } ]
  }
}
```

The contract has held up across several rounds of feature additions (tabs, look-through baskets, factor regression, anomaly detection, intraday QMLE). Adding a new analytical computation cleanly maps to either (a) a new field on an existing record, (b) a new top-level subtree, or (c) a new sub-record list under an existing record. The frontend treats unknown fields as no-ops and missing optional fields as feature-not-available, so producer and consumer can evolve at slightly different rates without coordinated deploy.

## 5. Methodology

The methodology section is the longest and most consequential — every other architectural choice serves the goal of making these computations transparent and auditable.

**Historical Simulation VaR** — empirical p-th quantile of trailing 1,000 daily returns. 1,000 days chosen because at p=0.01 the window contains 10 observations in the loss tail, the smallest count at which the empirical quantile is reasonably stable. Smaller windows yield noisy estimates; larger windows smear across volatility regimes. ES is the average of observations at or beyond VaR. No distribution assumption.

**EWMA VaR** — RiskMetrics-style exponentially weighted volatility, λ=0.94. Recursion `var_t = (1-λ)·r²_{t-1} + λ·var_{t-1}` run forward; most recent value is the conditional variance forecast. VaR and ES computed under Normal distribution. Known fat-tail miss at 1% — that's the deliberate baseline the other models are compared against.

**GARCH-t (1,1)** — Bollerslev (1986) conditional variance dynamics with **Student-t innovations** (`dist="t"` in the `arch` library, not Normal). Daily equity returns have empirical kurtosis well above the Normal's 3 (typically 5-10); Normal-innovation GARCH systematically understates 99% VaR by 30-60%. Student-t VaR uses the closed-form quantile of the standardized t with the estimated ν degrees of freedom; ES uses the EMF (2015) closed form. Falls back to EWMA on ARIMA convergence failure.

**GJR-t (1,1,1)** — GARCH-t plus an asymmetric (leverage) term: `var_t = ω + α·r²_{t-1} + γ·r²_{t-1}·I(r_{t-1}<0) + β·var_{t-1}`, still with Student-t innovations. Captures the leverage effect (negative shocks raise vol more than equally-sized positive shocks) on top of heavy-tailed innovations.

**EVT (Generalized Pareto)** — peaks-over-threshold. Losses exceeding the 90th-percentile threshold fit to a GPD via MLE; VaR computed analytically from GPD parameters. The only model in the suite that does not assume a particular *body* distribution at all — it models the tail directly. Tends toward over-conservatism on cleaner data; that's a feature for risk management but a calibration failure in formal backtests.

**Hill estimator** for tail-index α — losses sorted descending, α = k / Σ ln(L_i / L_k) where k ≈ √n with a minimum of 10. Lower α = fatter tails. Broad equity indices typically 3-4; individual stocks and crypto often 2-3.

**Component VaR** — portfolio's parametric VaR decomposed into per-holding contributions via EWMA covariance: `Component_i = w_i · (Σw)_i / σ_p · z · V`, where Σ is the EWMA covariance, w is weights, σ_p is portfolio vol, z is the Normal 1% quantile. Sum of components exactly recovers portfolio EWMA VaR by construction. Negative components indicate hedges.

**Backtesting** — Kupiec (1995) UC + Christoffersen (1998) IC on 504-day eval windows. For each day, the VaR forecast uses only the prior 1,000 days (strict OOS). Kupiec compares observed exception rate to expected p; Christoffersen tests independence of the violation indicator via a two-state Markov chain. Directional verdict from the two p-values: CALIBRATED, UNDER-EST, OVER-CONSERV, CLUSTERED. **All five models** are backtested. GARCH/tGARCH backtests use warm-started MLE — each refit's starting parameters are seeded from the previous fit's converged values, reducing iterations from 20-50 down to 3-10. Cached separately at `backend/cache/garch_backtests.json` and refreshed only on `RISKLENS_FULL_BACKTEST=1`.

**Historical scenarios** — replay actual prices over defined date ranges; portfolio P&L from re-weighted available tickers (re-normalizing if some tickers didn't exist in the date range). Fully data-driven, no shock vector or model assumption.

**Hypothetical scenarios** — per-asset shock vectors. Each scenario has a `shocks` dict mapping ticker → expected % move conditional on the scenario. Aggregation is `portfolio_pnl = Σ shocks_i · normalized_weights_i`. Per-asset (not factor-driven) specification; the linearity is intentional — it makes the contribution attribution exact and visible. Trade-off: shock vectors encode an implicit correlation assumption rather than computing it.

**Live recession probability** — Estrella-Trubin (2006) NY Fed probit model: `P(recession in 12mo) = Φ(-0.5333 - 0.6330·spread)`, where spread is the current 10Y - 3M Treasury yield difference. Yields from yfinance (`^TNX`, `^IRX`). Surfaced on the US Recession hypothetical scenario card.

**Intraday correlation** — 5-minute and 15-minute SPY-TLT log-return correlation per trading day, last 60 days (yfinance free-tier limit). Each daily value is its own statistically meaningful estimate (n ≈ 78 at 5-min, 26 at 15-min). The signal is consecutive same-sign streaks under a null of zero true correlation. **Two estimators are computed and toggleable on the frontend:**

- **Naive realized correlation** — standard sum of cross-products / sqrt of variance products.
- **QMLE polarization** — Aït-Sahalia, Fan & Xiu (2010) polarization identity `Cov(X,Y) = [IV(X+Y) − IV(X−Y)]/4`, with each leg's integrated variance estimated via Xiu (2010) univariate quasi-MLE on the MA(1) representation of noisy log returns. Recovers latent integrated correlation under microstructure noise. For SPY-TLT at 5–15m bars the difference is small (~0.05) because both are extremely liquid; the option exists as a robustness check.

**Fama-French + Momentum factor regression** (Sector Spotlight tab) — OLS on excess returns over the trailing 252 trading days. Six factors: Mkt-Rf, SMB, HML, RMW, CMA, Mom — Carhart (1997) extension of the Fama-French (2015) five-factor model. Outputs per-factor β with HC-naive t-stats and p-values, R², alpha (intercept) with t-stat and p-value, and vol decomposition (total, factor-attributable, idiosyncratic). Daily factor data from Ken French Data Library, fetched live with disk cache fallback at `backend/cache/ff_carhart_daily.csv`.

**Anomaly detectors** (Sector Spotlight tab) — three detectors run on each sector ETF's return series:

- **Standardized z-score** — `z_t = (r_t − μ_{60d}) / σ_{60d}`, flag |z| ≥ 3. Baseline outlier detection.
- **Two-sided Page CUSUM** — on standardized returns. `S^+_t = max(0, S^+_{t-1} + z_t − k)`, `S^-_t = min(0, S^-_{t-1} + z_t + k)`, allowance k=0.5, threshold h=5 in standard-deviation units. Catches sustained mean shifts the z-score misses.
- **GARCH-residual outlier** — `z̃_t = r_t / σ̂_t` where σ̂_t is the conditional std from a GJR-t-GARCH fit to the full series. Flag |z̃| ≥ 3. Captures days the conditional vol model didn't anticipate (already-vol-adjusted surprises).

Each detector's time series + the flagged-anomaly index are shipped to the frontend; the panel renders the price chart with markers + three stacked detector subpanels + a recent-flagged-dates list.

## 6. Active-fund look-through architecture

Active equity ETFs (CGGO, DWLD) are modeled as look-through baskets of their top-25 disclosed holdings, not as single-asset NAV series. The architecture:

1. **Sponsor disclosure file** dropped into `ext-data/` (xlsx for CGGO from Capital Group's daily-holdings download; csv for DWLD from Davis Advisors).
2. **`backend/preprocess_holdings.py`** parses the file, normalizes security names (strips boilerplate suffixes like "COMMON STOCK USD.01"), maps tickers to yfinance-friendly form (Korean A-prefix → `.KS`, Bloomberg-style suffixes → `.HK / .DE / .L / .SW / .AS`, ADR overrides for select international names), and emits a clean JSON at `backend/data/active_fund_holdings.json`. Run locally; the JSON is committed so CI doesn't need openpyxl.
3. **`run.py::_augment_active_fund_modes_with_holdings`** reads the JSON and rewrites each active-fund mode's `tickers` and `weights` to be the top-25 holdings (yfinance-mappable) re-normalized to sum to 1.0. The fund ticker itself is appended as a final reference row with no weight (so its NAV-based risk metrics are computed and displayed for comparison vs the basket, but it's not part of the portfolio aggregation).
4. **Pipeline runs unchanged** — `compute_mode` iterates the new tickers list, computes per-asset risk for each underlying, runs the basket through scenario/backtest/risk-history machinery. Component VaR decomposes basket risk per underlying.

**History filter in `compute_portfolio_row`** — tickers with < 252 days of return history (e.g., recent IPOs or relistings) are excluded from the basket aggregation and the remaining weights re-normalized. Per-asset risk rows still compute. Prevents one short-history holding from collapsing the basket's effective lookback to its inception.

**Coverage caveat surfaced on the Fund Holdings panel** — for CGGO the top-25 cover ~52% of fund weight; for DWLD ~83%. The remaining weight (cash, FX hedges, long-tail positions, unmappable foreign listings without ADRs) is excluded. Coverage % is computed at preprocessing time and shipped in the JSON so the frontend can render a yellow callout explaining the basket-vs-actual gap.

## 7. Frontend rendering

The frontend is structured around three top-level tabs. Each tab owns a set of analytical sections, each implemented as a self-contained component receiving a slice of the JSON.

**Tab 1 (Portfolio Risk):** `RiskTable`, `FundDisclosurePanel` (active-fund modes only), `PortfolioRiskChart`, `BacktestPanel`, `ScenarioPanel`.

**Tab 2 (Market Context):** `HistoricalChart` (S&P 500 + VIX), `CorrelationChart`, `MultiWindowCorrelationChart`, `IntradayCorrelationChart`.

**Tab 3 (Sector Spotlight):** `AnomalyDetectorPanel` (composite — renders the ticker dropdown, risk-profile card, factor-regression panel, stacked detector charts, and recent-anomalies list).

A reusable `Section` wrapper renders each section's title, plain-English question, and a collapsible "About this section" toggle that reveals the description and curated literature references.

The `CalendarHeatmap` component is reused across two contexts (intraday correlation calendar view + VaR exception calendar in the backtest panel). The `InfoTip` and `HoverTip` components handle tooltip rendering consistently.

Cross-component state is intentionally minimized. The active tab and the portfolio mode live at `App`; everything else is local to its section. This avoids the entire class of state-synchronization bugs that would emerge from a shared store, at the cost of some prop-drilling that hasn't become painful.

**Theme:** dark slate background, burnt-orange accent (`#d97706`), JetBrains Mono throughout. Deliberately distinct from electric-blue dashboards in the same space (extremerisk.org, generic Bloomberg-clone aesthetics). Single `:root` variable block in `index.css` controls the palette; recharts components use a small set of hardcoded hex values for chart line/gradient colors (Recharts doesn't read CSS variables at runtime).

## 8. Operational profile

Daily refresh runs unattended via GitHub Actions cron at 06:30 UTC weekdays. End-to-end wall-clock is approximately 4 minutes (Python install + Node install + JSON regeneration + frontend build + Pages deploy). Python compute itself is 90-120 seconds.

**Observed failure modes:**
- yfinance occasionally returns malformed responses for individual tickers (handled with per-ticker try/except in `fetch_data.py`).
- Yahoo throttles or temporarily blocks requests at high volume (no observed impact at our ~80 daily requests).
- GARCH MLE occasionally fails to converge for high-vol assets (handled with EWMA fallback).
- Ken French data fetcher fails when their site has maintenance (handled with local CSV cache fallback at `backend/cache/ff_carhart_daily.csv`).
- Recently-listed international tickers in active-fund baskets sometimes return < 250 days of history (handled by the history-filter in `compute_portfolio_row`).

None of these have caused a complete daily-refresh failure in observed runs.

**Storage:** the JSON output is approximately 1.2 MB uncompressed and gzips to ~120 KB on the wire. The cached GARCH backtest file is ~20 KB. The cached FF factor file is ~1.5 MB (CSV, 15k+ daily rows back to 1963). The entire repo + cache + JSON is under 10 MB.

**Compute profile:** single machine, single process. Memory peaks around 250 MB during the full-GARCH-backtest path. CPU is single-threaded; embarrassingly-parallel components are not currently parallelized because serial time is acceptable.

**Monitoring:** GitHub Actions surfaces job success/failure + execution time as standard UI. The frontend displays the `data_as_of` field in the header; if the daily refresh has been failing for multiple days, the displayed date will be stale — providing a passive freshness signal.

No application server, no live database, no cache layer beyond GitHub Pages's defaults. No operational state outside the JSON output, the GARCH cache, the FF factor cache, and the holdings JSON.

## 9. Scalability characteristics

**Per-asset risk computation** is O(N) in the number of unique tickers. Dominant per-asset cost is GARCH MLE fitting, ~0.5-1s per asset on commodity hardware. Current N ≈ 80 unique tickers (across all modes + bond proxies + sector ETFs); per-asset pipeline runs in ~60s. At N = 1,000 serial time becomes ~15-30 minutes; the pipeline is embarrassingly parallel and `multiprocessing.Pool` would scale near-linearly with workers.

**Per-portfolio computation** is O(P) in distinct portfolios. Dominant per-portfolio cost is the rolling backtest at ~1-2 minutes (GARCH cached) or ~5-8 minutes (full GARCH refit). At P = 5 (current) this fits in a daily window; at P = 100 it dominates and benefits from parallelization.

**Sector Spotlight views** are O(M) in number of sector ETFs (currently M = 14). Per-ticker cost is dominated by the GARCH-residual fit (~1-2s) and the FF regression (~50ms). Full anomaly phase runs in ~30-45s.

**JSON document size** is O(N·P + M·D) where D is the lookback days per anomaly view. Current 80 tickers × 5 portfolios + 14 ETFs × 504 days = ~1.2 MB uncompressed. At 10× scale the document would benefit from splitting into per-portfolio + per-ticker JSONs with frontend lazy-loading on mode/ticker selection; the mode-toggle and ticker-dropdown patterns already map cleanly to lazy fetching.

**Frontend memory** scales with the loaded JSON. At 1.2 MB, runtime JS heap after parsing is roughly 8 MB. Modern browsers handle without issue.

**Network:** single ~120 KB JSON fetch on page load, then no further network activity. CDN-cacheable; cache-busting on the JSON URL via content hash handles the daily refresh.

## 10. Risks and limitations

### Methodology limitations

The five-VaR-model approach is unusual; most risk dashboards pick one. The model-disagreement-as-signal philosophy is intentional but produces a denser visual surface than single-model dashboards.

Hypothetical scenario shocks are per-asset rather than per-factor. A factor-shock-propagation approach (Barra-style) would be more methodologically rigorous and more reusable across portfolios — the same "Taiwan invasion" factor shock would apply identically to any portfolio with the right factor loadings. The FF regression in the Sector Spotlight tab is a step in this direction but isn't wired into the scenario engine. Implementing scenario-via-factor-propagation requires bridging the FF loadings into the scenario aggregation code — non-trivial but well-scoped.

Backtesting is 1-day VaR only. Multi-period VaR (10-day, 1-month) is not implemented. Time-scaling under √N assumes Normal returns and breaks for fat tails; a rigorous multi-period implementation would either fit models to multi-day overlapping returns directly or use Monte Carlo path simulation. Either is a meaningful add.

The QMLE intraday correlation estimator addresses microstructure noise but only for SPY-TLT pairwise. A multivariate realized-kernel implementation (Barndorff-Nielsen, Hansen, Lunde, Shephard 2011) would generalize to the cross-asset correlation chart.

The FF factor regression is a single-period OLS (252-day lookback). Rolling refit with time-varying loadings would be more accurate but adds noise to the per-day output. Rolling beta charts could be added as a follow-up.

### Data limitations

yfinance occasionally returns inconsistent data. The implementation uses forward-filling for short gaps, which is correct for EOD analysis but would be wrong for sub-daily use.

The Treasury yield series (`^TNX`, `^IRX`) used for the recession probability has occasional unit inconsistencies; the implementation includes a heuristic correction (`_last("^TNX")` divides by 10 if the raw value exceeds 30, accommodating Yahoo's intermittent reporting in tenths-of-a-percent vs percent).

Active-fund holdings are static between user-triggered preprocessing runs. Capital Group publishes CGGO holdings daily but the dashboard reflects whatever's most recently committed to `ext-data/`. A more rigorous implementation would fetch holdings directly from sponsor APIs; the cleaner ones (DWLD daily CSV, CGGO daily xlsx download) are accessible but each has its own ingestion quirks.

International ticker mapping in `preprocess_holdings.py` is heuristic-based with a manual-override table for select names. New international holdings entering the top-25 may not map cleanly and would be silently excluded from the basket (still shown on the disclosure panel with no risk row). The override table needs maintenance as fund composition changes.

### Architectural limitations

The single-document JSON architecture has a soft ceiling around 5 MB on the wire. Past that, the frontend's parse-everything-up-front approach starts to feel slow. Mitigation (per-portfolio + per-ticker JSON split with lazy loading) is well-defined and easy to implement when needed.

The Python pipeline is serial. Parallelization is straightforward but adds operational complexity (more failure modes, harder log inspection); current scale doesn't justify it.

The frontend has no server-side rendering. Initial load shows a brief loading state before the JSON arrives. SSR is not architecturally hard but adds significant infrastructure (Node runtime, build pipeline complexity) for marginal benefit at current scale.

### Operational limitations

No alerting beyond GitHub Actions' built-in failure notifications. If yfinance returns garbage data for a week, the dashboard would silently display garbage. Only a user noticing the displayed `data_as_of` is stale would catch this. A simple health-check monitor against the JSON URL would close this gap.

No audit logging. Anyone with the URL can view the dashboard; there's no record of who viewed what when. Appropriate for a public OSS project, would not be appropriate for confidential data.

No automated regression tests on the math. The pipeline runs daily and produces output; if a methodology change introduces a bug the only catch is visual inspection. A small unit-test suite over canonical inputs would be valuable but doesn't exist yet.

## 11. Alternatives considered

**Vendor risk systems (Bloomberg PORT, MSCI BarraOne, Aladdin, FactSet)** — mature, factor-aware, regulator-trusted, far more comprehensive. Also expensive, proprietary methodology, designed for risk specialists. RiskLens is complementary, not a replacement: methodology-transparent visualization layer that can sit alongside vendor systems.

**Open-source quant frameworks (QuantLib, Pyfolio, vectorbt)** — more sophisticated risk math in places but different scope. QuantLib is fundamentally derivatives-pricing-first; Pyfolio is backtest-focused; vectorbt is systematic-strategy-tuned. The bespoke implementation in `risk_engine.py` is appropriate for what's needed.

**No-code BI tools (Tableau, PowerBI, Sigma, Looker)** — visualization layers, not computation layers. They can render numbers from somewhere else but cannot run their own Kupiec/Christoffersen tests or fit GARCH-t. For methodology-explicit use cases they're the wrong tool.

**Server-side rendering and live data** — would require an app server, database, cache, job queue, and corresponding operational infrastructure. The current batch-compute → static-artifact → static-render architecture is a deliberate simplification. Going live would be appropriate if intraday or sub-daily latency were a real requirement; it isn't.

**Database-backed data layer (Postgres, DuckDB, Parquet on S3)** — would replace the single-JSON file with structured storage and per-record queries. At current scale adds complexity without value; at 10× scale would become natural. The data model in §4 is already structured cleanly enough that a migration would be largely mechanical.

**Different chart library (D3, Plotly, Highcharts)** — Recharts was chosen for its React-native API and acceptable defaults. D3 offers more flexibility at significantly higher implementation cost; Plotly produces fine charts but adds a heavy bundle; Highcharts has a commercial license. Recharts has been adequate.

## 12. Open technical questions

**Barra-class factor model.** Current FF + Momentum is open-data and legitimate but lacks industry-within-country granularity, daily-refit factor covariance, and proprietary shrinkage. A Barra-like build adds factor-tilt risk decomposition, factor-driven scenarios, and per-holding factor exposures — multi-month project for a credible build.

**Multi-period VaR.** 1-day VaR is implemented; 10-day and 1-month aren't. Time-scaling by √N is the simple-but-fat-tail-broken approach; rolling-overlap fitting or Monte Carlo path simulation are the rigorous alternatives. Worth doing if the dashboard is ever used for regulatory-format reporting.

**Real-time intraday refresh.** Currently a daily batch artifact. Going intraday means rebuilding most of the engine and adding live data subscription infrastructure. Significant scope expansion.

**Holdings auto-ingestion for active funds.** CGGO and DWLD currently use the most recent xlsx/csv committed to `ext-data/`. Direct daily fetch from sponsor sites would be possible but each sponsor has its own quirks; not worth the brittleness at current scope.

**Custom portfolio composition.** Five modes are hardcoded; a "compose your own portfolio" UI would require persistent storage of user state and access control — both pushing the architecture out of static-site territory. Server-side rendering becomes appropriate at that point.

**Confidence interval bands on charts.** The risk trajectory chart shows a single point estimate per day. A 95% CI band would be more honest about estimation uncertainty but requires bootstrap resampling (expensive) or model-specific closed-form variance estimates (involved).

**Realized-kernel multivariate intraday covariance.** Extends QMLE polarization (currently SPY-TLT only) to the full cross-asset correlation chart. Methodologically interesting; significant new code (~500 lines) for limited visible payoff at our sampling frequencies.

**Active alerting.** Current "monitor displayed data_as_of and notice stale" is passive. An active monitor pinging the JSON URL daily and alerting on staleness would be small to add and worth doing if anyone actually relies on the dashboard.

## 14. Reproducing a daily run

```bash
git clone <repo>
cd risklens

# Backend
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.lock.txt

# Optional: regenerate active-fund holdings JSON from xlsx/csv in ext-data/
python preprocess_holdings.py

# Regenerate everything else
python run.py
# Writes: ../frontend/public/data/risk_output.json

# Optional: regenerate the GARCH backtest cache (slow, ~5 minutes)
RISKLENS_FULL_BACKTEST=1 python run.py

# Frontend
cd ../frontend
npm install
npm run dev
# Open http://localhost:5173
```

Production deploy is fully automated via the GitHub Actions workflow.
